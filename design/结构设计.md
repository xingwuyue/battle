# 战斗Demo结构设计

## 1. 整体架构规划

基于 Godot 的最佳实践，整个角色系统建议采用 **组件化（Component-based）** 和 **数据驱动（Data-driven）** 的设计思路。
其核心思想是将角色的“躯体（表现与物理）”、“大脑（控制逻辑）”和“灵魂（数值数据）”完全解耦。

---

## 2. 角色系统设计

为了满足玩家控制与 AI 控制的区分，我们通过为角色挂载不同的“大脑”来实现。

### 2.1 基础角色类 (Character)
所有的角色（无论玩家还是 NPC）都共用一套基础逻辑，我们创建一个基础场景 `Character.tscn`，根节点推荐使用 `CharacterBody2D`。
基础类只负责：
- 组装各个组件（状态机、属性组件、判定框等）。
- 执行物理移动。
- 处理特殊的高度系统（`height` 变量更新与 Y 轴视觉表现）。
- 播放对应状态的动画。

我们采用 **继承场景** 的方式来组织具体角色：
创建 `Player.tscn` 继承自 `Character.tscn`，再创建具体的 NPC 如 `Enemy_Melee.tscn` 继承自 `Character.tscn`。
- 这样每个角色可以有完全独立的节点结构调整（比如不同的 Sprite，不同的动画库），同时复用底层的状态机和判定逻辑。

### 2.2 控制器系统 (Controllers)
在基础角色的节点下，挂载一个控制器节点。控制器负责输出指令给角色的“状态机”。
1. **玩家控制器 (`PlayerController`)**: 
   - 监听设备的输入（WSAD、J、K、Space、Shift 等）。
   - 将输入转化为移动向量和状态切换请求（如按下 J 请求进入 `attack` 状态）。
2. **AI控制器 (`AIController`)**: 
   - 不监听设备输入，而是通过内部逻辑（简单的状态机或行为树）自动生成指令。
   - 例如：检测到敌人在攻击范围内 -> 自动触发进入 `attack` 状态的请求。

### 2.3 数据驱动：角色是否可以走 res？
**回答：完全可以，而且非常推荐使用 Resource（`.tres`）来管理角色。**

具体规划如下：
- **数据解耦 (`CharacterData.gd` 继承自 `Resource`)**：
  将设计文档中的“最大生命、最大体力、攻击、防御、暴击率、移动速度、阵营”等抽象为一个 Resource 脚本。
- **配置实例化**：
  在编辑器中，直接右键创建不同的 `.tres` 文件：
  - `player_data.tres`（阵营：玩家，高属性）
  - `enemy_melee.tres`（阵营：敌人，偏向血量与防御）
  - `ally_knight.tres`（阵营：友军，中等属性）
- **应用方式**：
  角色的基础类暴露出一个 `export(Resource)` 的变量。无论实例化多少个角色，只需把对应的 `.tres` 拖入，角色初始化时自动读取里面的数值即可。不同角色的区分就变成了数据文件的替换。

## 3. Demo 场景规划 (5个角色)

在一个测试场景（`DemoLevel.tscn`）中，放置 5 个基于基础角色场景派生出的实例：

1. **玩家 (1个)**：
   - 控制器：`PlayerController`
   - 数据：`player.tres` (阵营：玩家)
2. **敌人 (2个)**：
   - 控制器：`AIController` (配置为激进型 AI：索敌、追击、攻击)
   - 数据：`enemy_1.tres`, `enemy_2.tres` (阵营：敌人)
3. **同盟 (1个)**：
   - 控制器：`AIController` (配置为辅助型 AI：跟随玩家，攻击靠近的敌人)
   - 数据：`ally.tres` (阵营：友军)
4. **中立 (1个)**：
   - 控制器：`AIController` (配置为被动型 AI：平时 idle 或随机游走，受到攻击才反击或逃跑)
   - 数据：`neutral.tres` (阵营：中立)

## 4. 伤害与受击系统设计 (重点)

为了避免复杂的代码耦合，角色的伤害产生与接收采用 **Hitbox（攻击判定框）** 和 **Hurtbox（受击判定框）** 的模式。

### 4.1 伤害产生：Hitbox (攻击判定框)
- **挂载位置**：附加在角色的武器节点或攻击特效上。
- **触发时机**：由动画的状态机控制。在攻击动作的“出招”阶段激活碰撞，在“抬手”和“后摇”阶段关闭碰撞。
- **携带信息**：当它击中目标时，向外传递自身的数据（例如：当前角色的攻击力、是否触发了那 15% 的暴击、攻击者的阵营）。

### 4.2 伤害接收：Hurtbox (受击判定框)
- **挂载位置**：附加在角色的身体上，负责监听 Hitbox 的重叠事件。
- **阵营过滤**：Hurtbox 会读取当前角色的阵营，并与来袭 Hitbox 的阵营比对。如果属于同一阵营（或非敌对的中立），则忽略碰撞，从而防止误伤。

### 4.3 伤害结算与状态反馈流程
1. **碰撞发生**：敌人的 Hitbox 触碰到了玩家的 Hurtbox。
2. **数值计算**：
   - Hurtbox 获取 Hitbox 传递的攻击力。
   - 基础伤害 = `攻击力 - 自身防御力`（至少保底造成 1 点伤害）。
   - 如果 Hitbox 传递了“暴击标识”（根据15%概率算出），则 `最终伤害 = 基础伤害 * 1.5`。
3. **扣除血量**：Hurtbox 通知角色的属性组件扣除“当前生命”。
4. **状态打断**：
   - Hurtbox 向角色的状态机发送指令，**强制打断**当前状态（除非当前是 `die`）。
   - 状态机切入 `hit` 状态，角色播放受击动画、产生击退位移。
5. **死亡判定**：如果扣血后“当前生命 ≤ 0”，状态机直接切入 `die` 状态，角色死亡。

## 5. 项目目录结构建议

为了配合上述架构，建议使用以下文件结构：

```text
res://
├── components/         # 独立的功能组件
│   ├── state_machine/  # 状态机节点与各个状态脚本(idle, attack, hit等)
│   ├── hitbox/         # Hitbox 与 Hurtbox 场景及脚本
│   └── controllers/    # PlayerController, AIController 脚本
├── entities/           # 角色实体预制体
│   ├── base_character/ # 基础角色场景 (Character.tscn)
│   └── variants/       # 继承自基础场景的具体角色 (Player.tscn, Enemy.tscn等)
├── resources/          # 数据驱动文件
│   ├── character_data.gd   # 定义角色属性的 Resource 类
│   └── data_tres/      # 存放具体的 .tres 配置文件
└── scenes/             # 关卡与主场景
    └── DemoLevel.tscn  # 放置5个角色的测试场景
```
